# User Management System

A CRUD app that fetches user data from JSONPlaceholder's API, stores it in Redux Toolkit, and lets you view, add, edit, and delete users. All mutations happen locally in the Redux store; the API is only hit once on initial load.

Built with React 19, TypeScript, Redux Toolkit, and React Router v7. No UI libraries. CSS is Vanilla CSS.

---

## How it works

On first load, the app fetches 10 users from `https://jsonplaceholder.typicode.com/users` and stores them in a Redux slice. While that request is in flight, a spinner shows. If it fails, an error message appears instead. Once loaded, the user list renders as a grid of cards showing each person's name, email, and partial address.

From there you can view a user's full details on their own page (`/users/:id`), edit their info through a prefilled form (`/edit-user/:id`), add a new user (`/add-user`), or delete them straight from the card. The task spec explicitly says add, edit, and delete should be local-only: no POST, PUT, or DELETE requests go to the API. Redux handles all state changes in memory for the session.

The form component is shared between the add and edit pages. When adding, it renders with empty fields. When editing, the page pulls the existing user from Redux state and passes it to the form as `initialData`, which prefills every field. The form doesn't know or care whether it's being used for a new user or an existing one; it just collects data and calls `onSubmit` with whatever was entered.

Pages each get their own folder. Components shared across pages (`user-card.tsx`, `user-form.tsx`) sit in a top-level `components/` folder because they don't belong to any single page. Redux logic is split into two files: `store.ts` creates and configures the store, `user-slice.ts` defines the state shape, reducers, and the async thunk for the initial fetch.

## Routes

`/users` shows the full list with a loading spinner during the initial fetch. `/users/:id` shows one user's complete details. `/add-user` renders an empty form; on submit it generates an ID and adds the user to Redux state. `/edit-user/:id` renders the same form prefilled with the user's current data; on submit it updates the user in Redux state. No protected routes because the task doesn't require auth.

## Technical decisions

**Redux Toolkit over Context:** This project manages an array of users with loading and error states, plus four mutation operations. Redux Toolkit's `createSlice` handles all of that in one file with auto-generated action creators, and `createAsyncThunk` manages the fetch lifecycle (pending, fulfilled, rejected) without manual loading state management.

**Shared UserForm component:** The add page and edit page both need a form with the same fields. Instead of duplicating the form JSX in two files, `UserForm` accepts optional `initialData` (for prefilling during edits) and a generic `onSubmit` callback. The page components handle the Redux dispatch; the form handles the UI. This split keeps each file focused on one job.

**UserFormData as Omit<User, 'id'>:** When adding a user, the form collects name, email, phone, and address but not the id (which gets generated by the reducer). When editing, the id already exists and gets reattached by the edit page before dispatching `updateUser`. The `Omit` utility type makes this split explicit at the type level so the form never has to worry about IDs.

**Local-only mutations:** The task says not to make POST, PUT, or DELETE requests. So `addUser`, `updateUser`, and `deleteUser` are plain synchronous reducers, not async thunks. Only `fetchUsers` is async because it actually hits the network. All changes disappear on page refresh.

## Concepts practiced

Redux Toolkit: store configuration, createSlice, createAsyncThunk, extraReducers with the builder pattern, PayloadAction typing, useSelector with RootState, useDispatch with AppDispatch. React Router v7: nested routes with Layout and Outlet, dynamic `:id` params with useParams, programmatic navigation with useNavigate, index route redirect with Navigate. TypeScript: interfaces for data shapes, Omit utility type, keyof for type-safe form handlers, computed property names with type constraints. Component design: shared form component with optional initialData, separation between page-level dispatch logic and reusable UI components.
